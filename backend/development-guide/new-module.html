새로운 모듈 추가 ​

AxpFramework에 새로운 비즈니스 모듈을 추가하는 전체 가이드입니다.

개요 ​

새로운 모듈을 추가할 때는 표준화된 패키지 구조와 개발 패턴을 따라야 합니다. 이 문서는 모듈 생성부터 배포까지의 전체 과정을 안내합니다.

모듈 추가 프로세스 ​

1단계: 모듈 구조 설계 ​

새로운 모듈을 추가하기 전에 다음을 결정해야 합니다:

  • 모듈명: 예) hr, finance, inventory 등
  • 도메인: 모듈 내부의 기능 영역 (예: payroll, attendance, leave)
  • API 엔드포인트: REST API 경로 (예: /api/hr, /api/finance)

2단계: 패키지 구조 생성 ​

표준 패키지 구조를 따라 디렉토리를 생성합니다:

src/main/java/
└── {모듈명}/                        # 예: hr, finance
    └── {도메인}/                    # 예: payroll, attendance
        ├── controller/              # REST API 컨트롤러
        │   └── {모듈명}{도메인}Controller.java
        ├── service/                 # 비즈니스 로직
        │   └── {모듈명}{도메인}Service.java
        ├── mapper/                  # MyBatis Mapper 인터페이스
        │   └── {모듈명}{도메인}Mapper.java
        └── utils/                   # 모듈별 유틸리티 (선택)
            └── {모듈명}{도메인}Util.java

예시:

src/main/java/
└── hr/
    └── payroll/
        ├── controller/
        │   └── HrPayrollController.java
        ├── service/
        │   └── HrPayrollService.java
        └── mapper/
            └── HrPayrollMapper.java

3단계: 애플리케이션 설정 ​

ComponentScan 추가 ​

AxpBootApplication.java에 새 모듈을 추가합니다:

java
@SpringBootApplication
@ComponentScan(basePackages = {"axpFramework", "ess", "fcm", "hr"})  // 새 모듈 추가
public class AxpBootApplication extends SpringBootServletInitializer {
    // ...
}

MapperScan 추가 ​

MyBatis Mapper 인터페이스를 스캔하도록 설정합니다:

java
@MapperScan(
    basePackages = {
        "axpFramework.common.database.mapper",
        "ess.com.mapper",
        "hr.payroll.mapper"  // 새 모듈 Mapper 추가
    },
    sqlSessionFactoryRef = "sqlSessionFactory",
    sqlSessionTemplateRef = "sqlSessionTemplate"
)

4단계: 클래스 작성 ​

각 계층별로 클래스를 작성합니다:

  1. 컨트롤러 작성 - REST API 엔드포인트
  2. 서비스 작성 - 비즈니스 로직
  3. Mapper 작성 - 데이터베이스 접근

5단계: 리소스 파일 생성 ​

XML 매퍼 파일 ​

src/main/resources/mapper/{모듈명}/{도메인}/{모듈명}{도메인}Mapper.xml

예시:

src/main/resources/mapper/hr/payroll/HrPayrollMapper.xml

6단계: 설정 파일 추가 ​

필요한 경우 환경별 설정 파일에 모듈 설정을 추가합니다:

  • 설정 파일 추가

표준 패키지 구조 ​

전체 구조 ​

{모듈명}/
└── {도메인}/
    ├── controller/                 # REST API 컨트롤러
    ├── service/                    # 비즈니스 로직
    ├── mapper/                     # MyBatis Mapper 인터페이스
    └── utils/                      # 모듈별 유틸리티 (선택)

계층별 역할 ​

Controller ​

  • REST API 엔드포인트 제공
  • 요청/응답 처리
  • AxpBaseWebController 상속

Service ​

  • 비즈니스 로직 처리
  • 프로시저 호출 또는 Mapper 호출
  • 트랜잭션 관리

Mapper ​

  • 데이터베이스 접근
  • MyBatis XML 매퍼와 연동

실제 예시: ESS 모듈 구조 ​

현재 프로젝트의 ESS 모듈을 참고할 수 있습니다:

ess/
├── admin/                          # 관리자 기능
│   ├── controller/
│   ├── mapper/
│   └── service/
├── auth/                           # 인증 기능
│   ├── controller/
│   │   └── EssLoginController.java
│   ├── mapper/
│   └── service/
├── com/                            # 공통 기능
│   ├── controller/
│   │   └── EssComController.java
│   ├── mapper/
│   │   └── EssComMapper.java
│   ├── service/
│   │   └── EssComService.java
│   └── utils/
└── welfare/                        # 복지 기능
    ├── controller/
    ├── mapper/
    └── service/

공통 프레임워크 활용 ​

새로운 모듈에서는 공통 프레임워크의 기능을 활용할 수 있습니다:

데이터베이스 ​

  • 프로시저 호출: AxpCommDbService.executeProcedure()
  • Redis 관리: AxpCommRedisDbService

사용 예시:

java
@Resource(name = "AxpCommDbService")
private AxpCommDbService axpCommDbService;

Map<String, Object> requestBody = new HashMap<>();
requestBody.put("procedure", "P_YOUR_PROCEDURE");
requestBody.put("params", params);
Map<String, Object> result = axpCommDbService.executeProcedure(requestBody);

인증/보안 ​

  • JWT 토큰: AxpCommAuthService
  • 암호화: AxpCommAES128, AxpCommRsa

유틸리티 ​

  • 공통 유틸: AxpCommUtil
  • 파일 처리: AxpCommFileUtil
  • 검증: AxpCommValidationUtil

API 설계 규칙 ​

새로운 모듈의 API를 설계할 때는 다음 규칙을 따릅니다:

  • REST API 설계 규칙

주요 규칙:

  • 모든 API는 POST 메서드 사용
  • 요청 본문은 JSON 형식
  • 일관된 응답 형식 (status, message, data)
  • 적절한 HTTP 상태 코드 사용

인증/인가 처리 ​

새로운 모듈의 API에 인증이 필요한 경우:

  • 인증/인가 처리

설정 방법:

  • application-dev.yaml의 security.excluded-uris에 제외 URI 추가
  • 또는 JWT 필터를 통한 자동 인증

Swagger 문서화 ​

새로운 API를 Swagger에 문서화:

  • Swagger/OpenAPI 문서화
